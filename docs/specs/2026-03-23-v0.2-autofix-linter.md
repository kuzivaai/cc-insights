# cc-insights v0.2: Every Warning Has a Fix

**Date**: 2026-03-23
**Status**: Draft

---

## Position

The CLAUDE.md linting space now has 3+ competitors (cclint, AgentLinter, agnix) with 11-53+ rules each. cc-insights v0.1 has 8 rules. Competing on rule count is a losing game.

The gap: no existing tool auto-fixes its findings. They all show warnings. cc-insights v0.2 is the Prettier of CLAUDE.md. Install it, your file gets better, zero effort.

## Product Rules

1. Every rule has an auto-fix or a documented reason it doesn't.
2. 3-7 findings on a typical CLAUDE.md on first run. More triggers uninstall.
3. Under 200ms analysis time.
4. Zero configuration. No config file, no setup.
5. Suppression via `<!-- cc-insights-disable rule-id -->` inline comments.

### Rewrite

### New Rules

#### Tier 1: Auto-fixable

**`stale-import`**

Detects `@path/to/file.md` lines where the referenced file does not exist on disk.

- Category: `claudemd`
- Severity: `warning`
- Trigger: line starts with `@`, path does not resolve relative to the CLAUDE.md file location
- Auto-fix: remove the line
- Evidence: `{ reference, line, resolvedPath }`
- Implementation: extract lines matching `/^@(.+\.md)\s*$/`, resolve relative to CLAUDE.md directory, check existence via `ctx.fileExists()`

**`default-behavior`**

Detects instructions that match known phrases Claude follows without being told. These waste tokens with zero behavioral impact.

- Category: `claudemd`
- Severity: `info`
- Trigger: line matches one of the known default-behavior patterns (case-insensitive, fuzzy)
- Auto-fix: remove the line, show tokens saved
- Evidence: `{ matchedPhrase, line, estimatedTokens }`

Known phrases (initial set, not exhaustive):

```
"write clean code"
"write readable code"
"write maintainable code"
"follow best practices"
"handle errors appropriately"
"handle errors properly"
"handle errors gracefully"
"add comments where necessary"
"add comments where appropriate"
"use meaningful variable names"
"use descriptive variable names"
"be consistent with existing code"
"match existing code style"
"follow existing patterns"
"don't introduce bugs"
"avoid introducing bugs"
"test your changes"
"make sure tests pass"
"write tests for new code"
"use typescript types"
"add type annotations"
"keep code DRY"
"avoid code duplication"
"write efficient code"
"optimize for performance"
"follow SOLID principles"
"use proper error handling"
"validate input"
"sanitize input"
```

Match strategy: normalize whitespace, lowercase, check if any known phrase is a substring of the line. Lines inside code blocks are excluded.

**`filler-phrase`**

Detects padding phrases that add tokens without meaning.

- Category: `claudemd`
- Severity: `info`
- Trigger: line contains a filler phrase prefix
- Auto-fix: rewrite the line with the filler removed. "Always remember to use single quotes" becomes "Use single quotes"
- Evidence: `{ originalLine, rewrittenLine, line, tokensSaved }`

Filler prefixes:

```
"always remember to"
"make sure to"
"please make sure to"
"be sure to"
"please ensure that"
"ensure that you"
"it is important to"
"it's important to"
"you should always"
"you must always"
"please always"
"remember to always"
"don't forget to"
"keep in mind that"
"note that you should"
"be careful to"
"take care to"
```

Match strategy: check if a non-code-block line starts with (after stripping markdown list markers `- `, `* `, `1. `) any filler prefix. The auto-fix strips the prefix and capitalises the remainder.

**`redundant-config`**

Detects CLAUDE.md content that restates what is already in config files Claude reads directly.

- Category: `claudemd`
- Severity: `info`
- Trigger: CLAUDE.md contains a line that duplicates information from package.json, tsconfig.json, or .eslintrc
- Auto-fix: remove the redundant lines, add "See [config file] for [topic]" if the section becomes empty
- Evidence: `{ claudemdLine, configFile, configValue, line }`

Checks:
1. **package.json scripts**: if CLAUDE.md lists the exact same command as a package.json script (e.g., CLAUDE.md says `` `npm run build` `` and package.json has `"build": "tsc"`), flag it. The exception: if CLAUDE.md adds context beyond the command name (e.g., "Run `npm run build` before committing -- this also runs the type checker"), that is not redundant.
2. **tsconfig.json strict mode**: if CLAUDE.md says "use strict mode" or "TypeScript strict mode" and tsconfig.json has `"strict": true`, flag it.
3. **Linter config**: if CLAUDE.md says "use single quotes" and .eslintrc has a `quotes` rule set to `single`, flag it.

Implementation: read config files via `ctx.readFile()`, parse JSON, compare against CLAUDE.md content line by line.

**`non-md-import`**

Detects `@` import lines pointing to non-markdown files.

- Category: `claudemd`
- Severity: `warning`
- Trigger: line starts with `@`, path does not end in `.md`
- Auto-fix: remove the line
- Evidence: `{ reference, line, estimatedTokenCost }` (estimate token cost by reading the file if it exists)
- Message: includes estimated token cost if file exists. "This imports `src/main.ts` (847 lines, ~3,200 tokens) into every conversation."

#### Tier 2: Template insertion

**`no-structure`**

Detects CLAUDE.md files with zero markdown headings.

- Category: `claudemd`
- Severity: `info`
- Trigger: `parseClaudeMdSections()` returns a single section with heading `(no heading)`
- Auto-fix: analyse content and wrap in suggested sections. Heuristic: lines containing command patterns go under `## Commands`, remaining content goes under `## Conventions`.
- Evidence: `{ totalLines, estimatedTokens }`

**`missing-claudemd` (enhanced)**

When no CLAUDE.md exists but `package.json` is found, generate a starter.

- Auto-fix generates:

```markdown
# [name from package.json]

[description from package.json, if present]

## Commands

```bash
[each script from package.json as: npm run <name>  # <script value>]
```

## Conventions

[placeholder: "Add project-specific rules here."]
```

#### Tier 3: Warning only

**`settings-in-claudemd`**

Detects attempts to configure Claude Code settings in CLAUDE.md where they have no effect.

- Category: `claudemd`
- Severity: `warning`
- Trigger: line matches patterns like:
  - "you are allowed to" / "you have permission to" / "you can run"
  - "use opus" / "use sonnet" / "use haiku" / "prefer opus"
  - "always approve" / "auto-approve" / "don't ask for permission"
- Auto-fix: none (fix depends on which settings file to write to)
- Message: "Tool permissions belong in `.claude/settings.json`, not CLAUDE.md. See [docs link]."
- Evidence: `{ matchedPattern, line }`

**`position-risk`**

Detects critical instructions buried in the middle of long files where LLM attention is weakest.

- Category: `claudemd`
- Severity: `info`
- Trigger: a line containing emphasis markers (IMPORTANT, MUST, NEVER, CRITICAL, REQUIRED, DO NOT) is located in the middle third of a file with 100+ lines
- Auto-fix: none (moving rules changes document structure, needs human judgment)
- Message: "This rule is in the middle zone (lines X-Y) of a Z-line file. LLMs pay less attention to the middle. Consider moving critical rules to the top."
- Evidence: `{ line, totalLines, middleZoneStart, middleZoneEnd, markerFound }`

## Inline Suppression

Users can suppress specific rules on specific lines:

```markdown
<!-- cc-insights-disable stale-file-ref -->
`src/old-file.ts` is intentionally referenced for migration docs.
```

Or for a block:

```markdown
<!-- cc-insights-disable default-behavior -->
- Write clean, readable code
- Follow existing patterns
<!-- cc-insights-enable default-behavior -->
```

Implementation: before running rules, scan for `<!-- cc-insights-disable RULE -->` and `<!-- cc-insights-enable RULE -->` comments. Build a suppression map of `{ rule, startLine, endLine }`. Filter out insights that fall within suppressed ranges.

## VS Code Extension Changes

### Auto-fix on save

Add a `cc-insights.fixOnSave` setting (default: `false`). When enabled, all Tier 1 auto-fixes run on save. This is the Prettier model.

The setting is off by default because users need to see what the tool does before trusting it to modify files. After trust is established, they enable fix-on-save.

### "Fix all" command

Add command `cc-insights.fixAll` -- applies all auto-fixable rules to the current CLAUDE.md file. Shows a diff preview before applying.

### Diagnostic messages

Every diagnostic message must:
1. Say what is wrong (one sentence)
2. Say why it matters (token cost or behavioral impact)
3. Have a quick-fix action if auto-fixable

Example: `Stale reference: 'src/deleted.ts' -- file not found. This wastes ~30 tokens per conversation.`

Example: `Default behavior: "handle errors appropriately" -- Claude does this without being told. Removing saves ~8 tokens per conversation.`

### Performance

Target: under 200ms for a 500-line CLAUDE.md. All rules are string/regex operations except:
- `stale-file-ref`, `stale-command-ref`, `stale-import`: filesystem checks (async, parallel)
- `redundant-config`: reads config files (async, cached per session)

Cache config file reads. Only re-read when the file changes (use VS Code file watcher).

## Core Library Changes

### Removed exports

Remove `falseStarts`, `thinClaudemd`, `toolErrorPattern` from `ALL_RULES` and the rule registry. Keep the source files in the repo (don't delete) but remove from the default rule set. Consumers who imported them directly can still use them.

### New rule signature (unchanged)

All new rules follow the existing signature: `async (project: ProjectData, ctx: RuleContext) => Promise<Insight[]>`

### New field on Insight

```typescript
export interface Insight {
  // ... existing fields ...
  fix?: InsightFix;
}

export interface InsightFix {
  /** What the fix does, in plain English. */
  description: string;
  /** The replacement for the affected line(s). Empty string = delete the line. */
  replacement: string;
  /** Lines to replace (1-indexed, inclusive). */
  startLine: number;
  endLine: number;
}
```

The `fix` field is optional. Tier 3 rules (warning only) do not include it. Tier 1 and 2 rules always include it.

The VS Code extension maps `InsightFix` to `vscode.CodeAction` with `WorkspaceEdit`. The "fix all" command iterates all insights with `fix` defined, applies them bottom-to-top (to preserve line numbers).

### Suppression in the library

```typescript
export interface AnalyseOptions {
  // ... existing fields ...
  /** Lines to suppress, parsed from inline comments. If not provided, analyse() parses them from claudeMdContent. */
  suppressions?: Suppression[];
}

export interface Suppression {
  rule: string;
  startLine: number;
  endLine: number; // Infinity for single-line disable-to-end
}
```

`analyse()` parses `<!-- cc-insights-disable RULE -->` comments from `claudeMdContent` before running rules. Insights within suppressed ranges are filtered out.

## Testing Strategy

Each new rule gets a test file with:
1. Positive case: content that triggers the rule, assert insight with correct fix
2. Negative case: content that should not trigger
3. Suppression case: content with `<!-- cc-insights-disable -->` that should not trigger
4. Auto-fix verification: apply `fix.replacement` to the content, verify the rule no longer triggers
5. Code block exclusion: content inside fenced code blocks should never trigger text-analysis rules

Integration test: run `analyse()` on a fixture CLAUDE.md containing all rule triggers. Verify 3-7 findings (not more).
